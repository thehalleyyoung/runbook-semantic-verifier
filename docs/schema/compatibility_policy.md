# Runbook DSL schema compatibility policy

This policy defines how `docs/schema/runbook.schema.json`, parser behavior, and
generated action/condition descriptors evolve for users who validate runbooks in
editors, registries, and CI.

## Version scope

- The current DSL is versioned with the Python package version in
  `pyproject.toml`.
- The canonical schema is generated by `frv schema` and checked in at
  `docs/schema/runbook.schema.json`.
- A compatibility change covers the JSON Schema, parser validation, checker
  interpretation, exporter comments, and generated action semantics reference.

## Compatibility guarantees

Patch releases may:

- clarify descriptions, examples, and remediation text;
- add new diagnostics for inputs that were already invalid;
- improve checker precision without changing the accepted DSL shape.

Minor releases may add backward-compatible DSL features:

- new optional system fields;
- new actions, conditions, or optional parameters;
- new safety findings for newly modeled hazards.

Major releases are required for breaking changes:

- removing or renaming fields, actions, conditions, or parameters;
- changing a valid parameter type or numeric bound incompatibly;
- changing action semantics so an existing runbook means materially different
  operational behavior.

## Deprecation process

Any planned breaking DSL change should first ship as a minor release with:

1. the old form still accepted by the parser;
2. a warning-level diagnostic or documentation note naming the replacement;
3. migration guidance in this directory and examples where useful;
4. benchmark visibility when the change affects checked-in corpora.

The deprecated form should remain accepted for at least one minor release before
removal in a major release.

## Migration guidance

When adding DSL features, contributors should update all of the following in the
same change:

- typed descriptors in `src/runbook_verify/descriptors.py`;
- parser/entity validation and checker semantics;
- generated schema artifact and action semantics reference;
- at least one valid fixture or regression test;
- README or schema documentation when users need authoring guidance.

When a migration is needed, prefer accepting both old and new forms temporarily
and emitting precise diagnostics with path, line, field, severity, and
remediation.

## CI recommendation for users

Industry users should pin a package version in CI, check in the schema version
they validate against, and run `frv validate` before `frv check` so schema
incompatibilities fail fast without running state-space exploration.
